## 5. Integrating Security into Development Processes

FIASSE activity can be understood as three pillars:

- **Expectations**: security requirements, threat scenarios, acceptance criteria, and reference architectures set before code is written (Section 4.1, Section 4.2).
- **Implementation**: securable code driven by software engineering principles, with SSEM as the design language (Section 3, Section 4.3 through Section 4.6).
- **Assurance**: testing, review, and remediation applied strategically to reduce recurrence rather than to generate finding lists (Section 6).

These pillars are not domains in the sense of OWASP SAMM or similar maturity frameworks; they are simply a way to locate where a given FIASSE practice applies across the development process. The rest of this section describes how the pillars integrate with existing development workflows.
